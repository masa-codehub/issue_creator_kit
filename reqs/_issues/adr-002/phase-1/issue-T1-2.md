# Issue案: 既存ロジックの調査とメタデータ更新用正規表現の机上検証

- **Roadmap**: [roadmap-adr002-document-approval-flow.md](../_inbox/roadmap-adr002-document-approval-flow.md)
- **Task ID**: T1-2
- **Depends-On**: issue-adr002-T1-1.md
- **Status**: Draft

## 1. 目的と背景
既存の `create_issues.py` におけるメタデータ解析ロジックを理解し、新たに必要となる「メタデータ更新（置換）」のための正規表現パターンを確立します。
これは後続の設計（T1-3）およびテスト定義（T1-4）の基礎となります。

## 2. 実装指示 (Implementation Instructions)
- [ ] 既存の `src/issue_creator_kit/scripts/create_issues.py` を読み込み、メタデータ（`- **Key**: Value`）の抽出ロジックを特定・記録せよ。
- [ ] Markdown の行頭にあるキー（例: `Status`, `Date`, `Last Updated`）を正確に特定し、値を置換するための正規表現パターンを机上で作成せよ。
    - **要件**: 行頭の `- ` または `* ` に対応すること。キーと値の間の `**` や空白の揺らぎを考慮すること。
- [ ] 作成した正規表現が、以下のパターンにマッチし、かつ誤爆しないことを（オンラインツールや小さなスクリプトで）検証せよ。
    - `- **Status**: 提案中` -> マッチ
    - `* **Date**: 2025-01-01` -> マッチ
    - `- Status: 提案中` (太字なし) -> 対象外とするか検討
    - 本文中の `- **Status**: ...` -> マッチしないこと

## 3. 検証基準 (Definition of DoD)
- [ ] 既存ロジックの仕様が整理されていること。
- [ ] メタデータ置換用の正規表現パターンが確定し、検証済みであること。

## 4. 影響範囲と注意事項
- 実装（コーディング）は行わず、調査結果と正規表現パターンをドキュメント（コメント等）として残すこと。
