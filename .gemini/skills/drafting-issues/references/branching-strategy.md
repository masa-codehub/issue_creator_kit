# ブランチ戦略とタスク管理の考え方

`drafting-issues` スキルにおいて、ブランチ戦略（Branching Strategy）は、単なる作業場所の指定以上の意味を持ちます。それは「タスクの独立性」と「統合の安全性」を担保するための契約です。

## 基本原則

1.  **1タスク1ブランチ (One Task, One Branch)**
    - 各 Issue に対応する作業は、必ず専用の Feature Branch で行います。
    - これにより、並行して進む他のタスクと競合することなく、安全に実装・検証を行えます。

2.  **Base Branch は「安定した大地」**
    - `Base Branch`は、常に動作可能でテストが通る状態（Green）であることを前提とします。
    - タスクの開始時には、必ず `Base Branch` から分岐し、最新の状態を取り込みます。

3.  **命名規則による意味付け**
    - ブランチ名を見るだけで、「何のタスクか」「どの Issue に対応するか」が分かるようにします。
    - 形式: `<type>/task-<Task-ID>-<subject>`
      - `feat`: 機能追加
      - `task-T-01`: Issue ID (トレーサビリティの確保)
      - `update-schema`: 具体的な作業内容

## ワークフローにおける位置づけ

`drafting-issues` で作成される Issue ドラフトは、実装エージェントへの「発注書」です。この発注書の中でブランチ戦略を明記することで、実装エージェントは以下のように動くことができます：

1.  **迷わず開始**: 「どのブランチから切ればいいか？」を考えずに済む。
2.  **正しい場所で作業**: 誤って `main` にコミットしたり、関係ないブランチで作業したりするミスを防ぐ。
3.  **完了条件の明確化**: 「このブランチでの変更が、Base Branch にマージ可能であること」が完了の定義となる。
