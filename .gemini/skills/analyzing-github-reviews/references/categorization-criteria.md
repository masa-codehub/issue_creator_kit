# レビューコメント分類基準 (Categorization Criteria)

レビュアーからの指摘を以下の3つのカテゴリーに客観的に分類するための基準です。

## 1. Accept (受諾)
**「はい、その通りです」と回答すべきもの。**

- **基準:**
  - バグ、タイポ、未定義の挙動。
  - プロジェクトのコーディング規約（Styleguide, Lint等）への違反。
  - 明らかに可読性や保守性を向上させる代替案。
  - テストケースの不足や境界条件の考慮漏れ。
- **アクション:** 直ちにコードを修正し、必要であればテストを追加する。

## 2. Discuss (議論)
**「判断を仰ぐ、または詳細を確認する必要がある」もの。**

- **基準:**
  - 指摘の意図が不明確、または複数の解釈が可能な場合。
  - 提案された修正による副作用が懸念される場合。
  - トレードオフが発生し、設計思想（ADR）に関わる場合。
- **アクション:** 論点を整理し、ユーザー（開発責任者）に意見を求める。

## 3. Explain (説明)
**「現状のままであるべき理由がある」もの。**

- **基準:**
  - 指摘が既存の決定（ADR）やシステム境界（System Context）と矛盾する場合。
  - パフォーマンスや制約事項により、意図的に選択された実装である場合。
  - レビュアーの知識不足による誤認である場合。
- **アクション:** コードは変更せず、決定の背景や論拠を丁寧に返信する。

## 共通の行動規範
- **SSOTファースト:** 常に `docs/` 配下の公式ドキュメント（ADR, Spec）を正解として参照する。
- **ポジティブな受諾:** 改善提案には感謝を示し、資産（リファレンス等）への反映も検討する。
