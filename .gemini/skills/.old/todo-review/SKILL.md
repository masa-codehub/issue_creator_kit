# Todoレビュー (Todo Review)

作成された `.gemini/todo.md` を監査し、実行計画としての品質を保証する。
`todo-management` で規定された「Atomic Decomposition」の基準を満たしているかチェックする。

## プロセス (Process)

`.gemini/todo.md` に記載された**全てのタスク**に対し、以下の観点で厳しくレビューを行う。
改善すべき点が見つかった場合は、単に指摘するだけでなく、**「具体的かつ詳細な改善提案」**（具体的なコマンド例、分割方法、検証手順の記述など）を可能な限り多く列挙する。

### 監査項目 (Audit Checklist)

1.  **原子性 (Atomicity):**
    - 各ステップは「1つのアクション」に分解されているか？
    - 複数の操作（例：「Aを変更してBも修正する」）が1つのステップに混ざっていないか？
    - **改善提案:** (例: ステップ2を「Aの変更」と「Bの変更」の2つに分割する。)

2.  **検証可能性 (Verifiability):**
    - すべてのステップに明確な `Verify` 手順（コマンドや確認基準）が記述されているか？
    - 検証手順は、アクションの成功/失敗を客観的に判定できるか？
    - **改善提案:** (例: 目視確認ではなく `grep "expected_string" file.txt` を検証コマンドに追加する。)

3.  **論理的連続性 (Continuity):**
    - ステップ間に論理的な飛躍がないか？（AをしてCをする前に、Bが必要ではないか？）
    - 最終ステップ完了時に、確実に `Goal` が達成される構成になっているか？
    - **改善提案:** (例: ステップ3の前に「設定ファイルのバックアップ」ステップを追加する。)

4.  **順序性 (Ordering):**
    - 依存関係（先にやるべきこと）は正しい順序で並んでいるか？
    - 「調査」→「計画」→「実装」→「検証」のサイクルが守られているか？
    - **改善提案:** (例: 実装ステップの前に、テストケース作成ステップを移動する。)

## アウトプット (Final Output)
各監査項目で列挙された改善提案の中から、**「改善することで抜け・漏れ・無理・無駄が確実に小さくなる」** 価値ある指摘のみを厳選（ピックアップ）し、最終的なレビュー結果として提示する。
※ 改善効果が見込めない些末な指摘は除外し、計画の確実性を高める修正点のみに集中させる。
全てのタスクが基準を満たしており、重要な改善点がない場合のみ、計画を承認（Approve）する。
