# 開発標準の整備 (Standardization)

開発チームの生産性とコード品質を向上させるためのガイドラインやルールを策定・文書化するプロセスです。

### Step 0: 課題の特定 (Observe)

1.  **現状の課題分析:**
    *   開発プロセスにおける手戻り、バグの傾向、オンボーディングのコストなどを分析し、標準化が必要な領域を特定する。
    *   `read_file` で既存の `docs/guides/` を確認し、情報の不足や陳腐化がないかチェックする。

2.  **チームの合意形成:**
    *   （必要に応じて）Issueなどで開発チームにヒアリングを行い、ルールの必要性や方向性について合意を得る。

### Step 1: ルールの策定 (Orient)

具体的なルールや手順を設計します。

1.  **コーディング規約:**
    *   命名規則、ディレクトリ構成、コードフォーマット、コメントの書き方などを定義する。
    *   言語標準（PEP8など）や既存の慣習（Linter設定）を尊重する。

2.  **ワークフロー定義:**
    *   環境構築手順、ブランチ戦略、コミットメッセージの形式、PRの作成・レビュー手順などを定義する。

3.  **技術選定の標準化:**
    *   使用するライブラリやツールの推奨バージョン、禁止事項などを定義する。

### Step 2: ガイドの作成 (Act)

策定したルールをガイドドキュメントとして作成・更新します。

1.  **ドキュメント作成:**
    *   `write_file(file_path="docs/guides/xxx.md")` を実行し、ガイドを作成する。
    *   **ポイント:**
        *   **具体的であること:** 抽象論だけでなく、具体的なコード例やコマンド例を示す。
        *   **実行可能であること:** 手順通りにやれば誰でも同じ結果になるように記述する。
        *   **Whyを記述する:** なぜそのルールが必要なのか（背景、メリット）を説明し、納得感を高める。

2.  **周知:**
    *   作成したガイドをチームに共有し、運用を開始する。

### テンプレート: 開発ガイド (docs/guides/xxx.md)

```markdown
# [ガイド名]

## 目的
(このガイドの目的と対象読者)

## 前提条件
(必要なツールや環境)

## 手順 / ルール

### 1. 項目名
説明...

```bash
(コマンド例)
```

### 2. 項目名
説明...

## FAQ / トトラブルシューティング
...
```
